```sh
# 将文件修改或者创建操作 提交到暂存区
# 可以使用这个命令对 修改过的文件进行操作
# 也可以对Untracked files进行操作， Untracked files指的是新创建的还没有提交到缓存区的文件
git add haha.txt
# 逆操作，将add的文件从缓存区取回到工作区
git reset HEAD haha.txt

# 删除文件，并提交到缓存区
# 本地仍然需要使用, 只是不希望这个文件被版本控制，使用--cache
git rm index.html --cache
# 本地也不需要了
# 逆操作，取回到暂存区
git reset HEAD index.html

############################## 创建本地仓库
git init
# 之创建了 .git文件夹下的文件和文件夹
git init --bare

# 配置用户名和邮箱
git config user.name 'zhangsan'
git config user.email 'zhangsan@163.com'

############################## 基本操作
git add
git status
git commit -m ''
git push origin master

############################## 查看版本
git log # 没有回退记录
git reflog # 精简，有回退记录
# 显示的结果是按照时间倒序，格式为：

    init message


############################## 版本回退
git reset
### 方式①
HEAD表示当前最新版本
HEAD^表示当前最新版本的前一个版本
HEAD^^表示当前最新版本的前两个版本，以此类推...
HEAD~1表示当前最新版本的前一个版本
HEAD~10表示当前最新版本的前10个版本，以此类推...
#
git reset [--head] HEAD~2
### 方式2 指定版本hash
git reset [--hard] 版本号hash

若当前版本为A
git reset 回了B，之后：
A相对于B增加的文件会变成Untracked files
A相对于B修改的文件是未提交的modified状态
A相对于B删除的文件是未提交的deleted状态

############################## 代码撤销
# 如果已经执行了git add或者git rm操作
git reset HEAD 文件
git checkout 文件
# 如果还没有执行git add或git rm操作
git checkout 文件 # 这样就将刚才的修改discard了

##### 那么？？ 如果已经commit了怎么办？先 git reset 回退版本呀

问题：将xxx文件回退到hash为hashvl的版本
git reset hashvl
git checkout xxx
############################## 对比版本
##### 对比版本库与工作区
git diff HEAD
# 只对比指定的文件
git diff HEAD -- 2.txt

##### 对比版本库，即两次不同的提交
git diff HEAD HEAD^ -- 2.txt


############################# 配合远程仓库协作开发
### 本地添加标签
git tag -a 标签名 -m '标签描述'
# 如
git tag -a v1.0 -m 'version 1.0'
## 要注意添加完标签之后，要push
git push origin 标签名 (注意是标签名)
# 例：
git push origin v1.0


#### 删除本地和远程标签
# 删除本地标签
git tag -d 标签名
# 删除远程仓库标签
git push origin --delete tag 标签名

################## 分支相关
### 查看当前分支
git branch
### 创建并切换到`dev`分支
git checkout -b dev

# 假如进行添加 dev.txt
touch dev.txt
git add .
git commit -m 'add dev.txt'
git push origin dev

### 切换到master分支，并将dev分支合并过来
git checkout master
git merge dev ## 合并之后所有文件都是提交状态
git push
### 这时其他在主分支开发的同学可以拉代码了
git pull

#### bug分支
# 在dev分支上编辑代码, v1.0版本突然发现bug
# 记录dev分支未提交的修改：保留现场
git stash
# 切换到master分支
git checkout master
# 新建fixbug分支，并v1.0版本的代码下到fixbug分支
git checkout -b fixbug v1.0
# 修复线上v1.0版本bug，最后push更改
# 合并fixbug分支到master分支，并推送到远程仓库
  # 切换到master分支
  git checkout master
  # 合并fixbug分支到master分支
  git merge fixbug
  # 推送合并到远程仓库
  git push
### 经过上面的操作之后，v1.0的bug解决了，并且合并到了master分支上
# 打标签v1.1，发布v1.1版本
git tag -a v1.1 -m 'version 1.1'
git push origin v1.1

### 继续回到dev上进行开发
# 切换到dev分支上
git checkout dev
# 恢复
git stash list
git stash pop
## 将fixbug分支修改bug的内容也合并过来
  # 切换到dev分支
  git checkout dev
  # 合并fixbug分支到dev分支
  git merge fixbug
  # 推送
  git push
## 这样所有的分支存在的bug都修改了

在合并分支时，默认采用Fast Forward模式
该模式没有分支合并的版本记录
如果需要在合并分支时记录版本需要禁用Fast Forward模式
git merge --no-ff -m '合并描述' 分支名

Git很智能，如果别人新的提交没有涉及到你的新的修改，那么就会直接merge过来，如果涉及到你的修改,那么在pull时会：

error: Your local changes to the following files would be overwritten by merge:
        xxxxx
Please commit your changes or stash them before you merge.
Aborting

这样新的代码就不会merge过来，而解决的方法就是上面的提示，先stash自己的本地更改，然后pull拉取别人提交的新代码,再通过stash pop将自己刚才的本地修改merge到文件中:
git stash
git pull origin master
git stash pop

另外如果不想要自己的更改了，而是想要直接pull他人的代码，可以直接将自己的本地修改还原：
git checkout  文件
# 然后pull
git pull origin master
```
